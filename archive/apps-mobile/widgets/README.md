# Widgets — iOS WidgetKit + Android App Widget

Three home-screen widgets:

| ID | Size | Surface | Refresh |
|----|------|---------|---------|
| `today-leads` | small (2x2) | Today's lead count + delta vs yesterday | 15 min, plus push-driven refresh on new lead |
| `active-campaigns` | medium (4x2) | Live funnels and their status | 30 min |
| `conversion-ticker` | large (4x4) | 24-hour conversion sparkline + last lead name | 15 min |

All widgets tap-through to the matching screen in the app via the
`funnelai://` deep link scheme:

| Widget | Deep link |
|--------|-----------|
| today-leads | `funnelai:///(tabs)/leads` |
| active-campaigns | `funnelai:///(tabs)/funnels` |
| conversion-ticker | `funnelai:///(tabs)` |

## iOS (WidgetKit)

WidgetKit can't run JS — it's a native Swift extension built on top of the
Expo project. We declare the widget extension in `eas.json` build config and
generate the Swift sources from this directory.

Status: native extension scaffolding lives in `ios/` (generated by
`expo prebuild` when CNG runs). The Swift sources are kept in
`./ios/FunnelWidgets/`.

## Android (Glance App Widget)

Glance is the Jetpack Compose-style API for app widgets. Sources live in
`./android/funnelwidgets/`.

## Data feed

Both platforms share a tiny widget service in `lib/widget-data.ts` (mobile
process) that hydrates a shared App Group container (iOS) / SharedPreferences
(Android) on every app foreground. Widgets read from that container — they
never make network calls themselves.

Payload shape:

```ts
type WidgetSnapshot = {
  leadCountToday: number;
  leadCountYesterday: number;
  activeFunnels: number;
  conversionsBy15m: number[]; // 96 buckets
  lastLead?: { name: string; capturedAt: string };
  refreshedAt: string;
};
```
